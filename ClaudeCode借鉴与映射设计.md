# Claude Code 借鉴与映射设计

## 一、文档目的

本文档用于明确：

1. `claude-code-sound` 与 `NovAI` 的关系
2. NovAI 要借鉴 Claude Code 的哪些核心实现思路
3. NovAI 明确不打算照搬哪些复杂能力
4. Claude Code 的哪些设计需要映射、改造后才能适配小说创作场景

这份文档的重点不是“复刻 Claude Code”，而是借它验证一种更成熟的产品形态：

`对话驱动的 AI 智能体 + 工具调用 + 文件落地`

---

## 二、两个项目的关系

### 2.1 Claude Code 参考项目的角色

`/Users/honlnk/project/claude-code-sound`

这个项目在 NovAI 中的定位是：

- **参考实现**
- **交互逻辑样板**
- **智能体式产品形态的学习对象**

它不是：

- NovAI 的直接上游
- NovAI 的内核依赖
- 一个要被完整复刻的目标产品

### 2.2 NovAI 的目标产品形态

NovAI 的目标不是做一个“小说主题聊天工具”，也不是做一个“只有生成按钮的工作台”。

更准确地说，NovAI 想做的是：

> 一个面向小说创作者的、可视化的、对话驱动的 AI 写作智能体工具。

它的核心工作方式应当是：

1. 用户通过自然语言表达创作或修改意图
2. AI 根据当前视觉上下文与项目状态理解“用户想操作什么”
3. AI 通过工具读取或修改对应文件
4. 结果落到章节、提示词、要素等本地 Markdown 文件中
5. 对话区承担协作与反馈，文件区承担内容呈现与最终存储

---

## 三、NovAI 要借鉴的核心，不是表面 UI

Claude Code 最值得借鉴的不是“终端长什么样”，而是它的底层工作逻辑。

NovAI 需要借鉴的核心有四点：

### 3.1 对话不是文本输出器，而是智能体控制面板

在 Claude Code 中，对话的作用不是单纯输出长文本，而是：

- 接收用户目标
- 驱动 AI 进入多步执行
- 展示工具调用和执行摘要
- 在必要时继续追问或请求确认

NovAI 也应该采用这个方向。

对于小说创作来说，对话区应该主要承担：

- 用户表达“写什么”“改哪里”“怎么改”
- AI 返回操作摘要、思考结论、需要澄清的问题
- 用户继续细化创作意图

而不是把完整正文长期堆在聊天消息里。

### 3.2 会话核心应当是“持续运行的 Agent Loop”

NovAI 不应长期停留在：

`输入一段 instruction -> 调一次模型 -> 返回一段文本`

而应逐步转向：

`用户提出目标 -> AI 自动决定读取什么 -> 分析 -> 修改/写入文件 -> 返回结果摘要`

也就是说，NovAI 未来的核心应该是：

- 一个持续存在的会话对象
- 一套可调用的工具
- 一个围绕工具不断推进任务的查询循环

### 3.3 文件是主存储，对话只是协作层

这一点 Claude Code 很强，NovAI 也必须坚持。

对 NovAI 来说：

- 章节内容保存在 `chapters/`
- 提示词保存在 `prompts/`
- 要素保存在 `elements/`
- 对话历史不是小说内容的主存储位置

### 3.4 消息流应当是“操作轨迹”，不是纯聊天记录

NovAI 后续的会话消息不应只有：

- 用户文本
- AI文本

还应逐渐包含这些更结构化的消息：

- 当前操作目标
- 文件读取摘要
- 文件写入结果
- 要素提取结果
- RAG 检索结果摘要
- 上下文压缩摘要
- 错误与阻塞提示

---

## 四、NovAI 明确不打算照搬的部分

Claude Code 很强，但它的大量能力对 NovAI 来说是无关复杂度。

当前明确**不照搬**的内容包括：

- 多 agent / swarm 协作
- MCP 生态
- LSP 集成
- IDE Bridge
- Web Search / Web Fetch
- Task 系统
- Team / Peer 通信
- Cron / Remote Trigger
- 复杂权限模式与组织策略系统
- 终端沉浸式 UI
- 面向编程的大量 slash commands

NovAI 第一阶段只需要保留最小工具集。

## 五、NovAI 的最小工具集

NovAI 第一阶段的工具集应控制在最小必要范围内：

- `ReadFile`
- `EditFile`
- `CreateFile`
- `RenameFile`
- `DeleteFile`
- `ListDirectory`
- `FindFiles`
- `RagSearch`

其中项目文件工具借鉴 Claude Code 的工具化思想，`RagSearch` 是 NovAI 的自定义核心能力。

### 5.1 不做 Bash

NovAI 后续不开发通用 `BashTool`。

Claude Code 中通过 Bash 完成的常见能力，在 NovAI 中应拆成受控专用工具：

- `ls` 场景由 `ListDirectory` 承担。
- `find / Glob` 场景由 `FindFiles` 承担。
- `rm` 场景由 `DeleteFile` 承担，并移入 `.novel/trash`。
- `mv` 场景由 `RenameFile` 承担。
- `grep / rg` 场景未来由 `RagSearch` 承担。

这样可以保留 Claude Code 的工具化工作流，同时避免把通用系统命令权限暴露给小说写作 Agent。

### 5.2 ReadFile

用途：

- 读取当前目标章节
- 读取当前提示词文件
- 读取要素文件
- 读取配置或相关上下文文件

这是 NovAI 智能体的高频核心工具之一。

### 5.3 CreateFile

用途：

- 创建新章节
- 写入新的要素文件
- 写入新的提示词文件

`CreateFile` 只负责新建，目标已存在时失败。完整覆盖类能力暂缓，未来如果新增 `WriteFile`，必须建立在读状态、过期检查和确认 UI 之上。

### 5.4 EditFile

用途：

- 修改已有章节
- 修改已有提示词
- 修改已有要素文件

### 5.5 RenameFile / DeleteFile

用途：

- 重命名章节、提示词或要素文件
- 将文件移动到新的项目目录
- 把废弃草稿移入 `.novel/trash` 回收站

这两个工具替代 Claude Code 中 `mv / rm` 的常见用法。

### 5.6 ListDirectory / FindFiles

用途：

- 查看目录直接结构
- 按路径模式查找文件
- 帮助模型在读取正文前先定位目标文件

这两个工具替代 Claude Code 中 `ls / Glob / find` 的常见用法。

### 5.7 RagSearch

这是 NovAI 相比 Claude Code 最关键的新增工具。

用途：

- 根据用户当前意图做语义检索
- 召回人物、地点、情节、时间线、世界观等要素
- 为生成或修改提供相关背景信息

结论：

NovAI 的主检索能力不应该像 Claude Code 那样主要依赖 `grep / rg / bash search`，而应该优先依赖 `RAG 语义检索`。

## 六、提示词体系：不采用 CLAUDE.md 机制

Claude Code 的 `CLAUDE.md` 自动注入机制，本质上是“项目级规则文件自动注入会话上下文”。

NovAI 不准备直接照搬这套机制，原因是：

1. NovAI 已经有自己的提示词体系
2. 提示词在 NovAI 中本来就是正式数据，而不是隐式规则补丁
3. NovAI 需要的是“可编辑、可预览、可保存”的提示词工作流

因此，NovAI 的上下文注入应当来自它自己的文件体系，例如：

- `prompts/system.md`
- `prompts/scenes/*.md`
- 后续可能增加的功能提示词模板

这意味着：

- NovAI 要借鉴的是“上下文可由本地文件驱动”这个思想
- 但不采用 `CLAUDE.md` 作为具体实现

## 七、NovAI 的 UI 交互要比 Claude Code 更可视化

Claude Code 是终端工具，用户主要通过文本描述来指定操作目标。

NovAI 的用户不是程序员，而是小说创作者，所以必须增加可视上下文。

### 7.1 当前预览对象应成为默认操作目标

这是 NovAI 很重要的一条交互原则：

> 用户当前正在预览什么，AI 默认就优先把它视为当前操作目标。

例如：

- 当前正在预览某一章正文  
  AI 默认认为用户想修改这个章节

- 当前正在预览某个场景提示词  
  AI 默认认为用户想修改这个提示词

- 当前正在预览人物要素  
  AI 默认认为用户接下来提到的人物设定调整，很可能就是改这个文件

### 7.2 视觉上的分区，不等于底层流程不同

在 UI 上，用户会感觉自己在：

- 修改章节
- 修改提示词
- 修改要素

但底层其实可以统一为同一种智能体工作流：

1. 根据当前预览态确定默认目标文件
2. AI 用 `FileRead` 获取内容
3. 必要时结合 `RagSearch` 获取相关语义上下文
4. AI 分析用户意图
5. 用 `EditFile`、`CreateFile`、`RenameFile` 或 `DeleteFile` 落地
6. UI 更新预览，并在对话区显示操作摘要

所以：

- UI 可以更符合创作者心智
- 内核仍然保持统一

### 7.3 用户不需要精确表达“工具名”

NovAI 不应要求用户像程序员一样说：

- 修改 `prompts/scenes/scene-001.md`
- 编辑 `chapters/第003章-密境遇险.md`

更合理的体验是：

- “把这一章结尾改紧张一点”
- “把这个场景提示词改得更偏悬疑”
- “把主角在这段里的情绪写得更克制”

系统再根据当前预览状态和上下文判断默认操作对象。

## 八、NovAI 中的 RAG 不只是附加功能，而是核心工具能力

Claude Code 的上下文获取很强，但主要依赖：

- 文件读写
- 文本搜索
- 命令执行

NovAI 的差异化核心在于：

> 它要让 AI 能通过语义检索理解“故事相关性”，而不仅仅是字符串匹配。

因此，RAG 在 NovAI 中不是边角料，而是主能力之一。

### 8.1 RAG 的职责

RAG 主要用于：

- 召回相关人物
- 召回地点与世界设定
- 召回情节线索
- 召回上一章节中与当前任务最相关的要素

### 8.2 RAG 在智能体工作流中的位置

NovAI 的 Agent Loop 未来应当允许出现这样的步骤：

1. 用户提出修改或创作目标
2. AI 判断需要补充背景信息
3. AI 调用 `RagSearch`
4. AI 读取最关键的目标文件
5. AI 结合目标文件 + RAG 结果进行分析
6. AI 执行修改或生成

这和 Claude Code 有本质区别：

- Claude Code 是“工具型编码智能体”
- NovAI 应该是“带语义检索能力的小说写作智能体”

## 九、对 Claude Code 的映射关系

下面是一个简化映射表。

| Claude Code 中的概念 | NovAI 中的映射 | 是否直接照搬 |
|:---|:---|:---|
| QueryEngine / query loop | NovAI 会话引擎 / Agent Loop | 借鉴思路，不直接照搬 |
| Tool system | NovAI 最小工具层 | 借鉴 |
| FileRead / Write / Edit | 文件操作能力 | 拆分为 `ReadFile / EditFile / CreateFile`，`WriteFile` 暂缓 |
| Bash | 辅助命令工具 | 不保留，改用专用项目文件工具 |
| CLAUDE.md 注入 | `prompts/system.md` / `prompts/scenes/*.md` 注入 | 改造替换 |
| grep / rg 文本检索 | `RagSearch` 语义检索 | 不照搬，改为 NovAI 核心能力 |
| 终端文本式目标指定 | 可视化预览态 + 默认目标文件 | 不照搬，改为可视化交互 |
| 大量 slash commands | 少量面向创作的自然语言操作 | 不照搬 |
| 多 agent / swarm | 单 agent 为主 | 当前不做 |
| IDE / 编程工具生态 | 小说项目工作台 | 完全替换 |

## 十、对当前 NovAI 文档体系的修正含义

这次明确方向之后，NovAI 后续文档需要逐步修正一个关键认知：

以前更像是在描述：

`一个有聊天区的小说工作台`

现在更准确的方向应该是：

`一个以可视化文件工作区为外壳、以 Agent Loop 为核心、以 RAG 为差异化能力的小说创作智能体工具`

这会影响后续几类文档：

- 产品规划
- AI 功能需求
- UI 设计
- 技术架构设计
- 开发前最小契约

---

## 十一、对当前实现的直接启发

结合现有代码状态，当前最值得推进的不是继续加按钮，而是开始补“会话与工具”这一层。

第一批最关键的新增能力应当是：

1. 会话状态模型  
   不再只有 `instruction` 和 `result`，而是有结构化消息列表

2. 最小工具协议  
   当前支持 `ReadFile / EditFile / CreateFile / RenameFile / DeleteFile / ListDirectory / FindFiles`，后续接入 `RagSearch`

3. 当前预览目标注入  
   把“当前选中的文件或分组”作为会话默认上下文的一部分

4. 修改类工作流统一化  
   章节、提示词、要素在底层统一为“读取目标 -> 分析 -> 修改 -> 写回”

5. RAG 工具化  
   不只是测试页按钮，而是让 AI 能在会话中主动调用

---

## 十二、一句话结论

NovAI 不应该复刻 Claude Code 的全部能力，但应该借鉴它最本质的产品逻辑：

> 让 AI 以对话为入口、以工具为手段、以文件为结果、以 RAG 为增强，帮助小说创作者完成真实的写作与修改工作。
