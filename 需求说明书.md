# NovAI（诺艾）- 需求说明书

> [!info] 本文档定位
> 详细描述 NovAI 各功能模块的具体需求，包括输入输出、交互逻辑、边界条件等，供开发团队对齐理解。
> 产品规划文档见 [产品规划](NovAI产品规划.md)。

## 一、概述

### 1.1 用户角色

NovAI 不设用户系统，使用者即作者。一个文件夹就是一个小说项目，不同项目之间数据完全隔离。

### 1.2 使用场景

| 场景 | 描述 |
|:-----|:-----|
| 新建项目 | 创建一个小说项目文件夹，配置模型、设定系统提示词 |
| 日常创作 | 输入指令让 AI 生成内容，AI 自动写入文件并提取要素 |
| 提示词调教 | 与 AI 对话来编写和优化系统提示词 / 场景提示词 |
| 内容校对 | 让 AI 检查章节一致性和要素准确性 |
| 章节整理 | 让 AI 按规范对散乱的章节文件进行拆分、合并、重命名 |

## 二、项目配置需求

### 2.1 模型配置

**目标**：让用户接入任意兼容 OpenAI 格式的大模型 API。

#### 用户操作

1. 进入模型配置页面
2. 填写以下信息：
   - **API 地址**（必填）：模型服务的 Base URL
   - **API Key**（必填）：认证密钥
   - **模型名称**（可选）：指定使用的模型，不填则使用 API 默认模型
3. 点击"测试连接"验证配置是否可用
4. 保存配置

#### 输入/输出

| 项目 | 说明 |
|:-----|:-----|
| 输入 | API 地址、API Key、模型名称 |
| 输出 | 配置保存成功 / 连接测试结果（成功/失败 + 原因） |
| 存储 | `novel.config.json` |

#### 边界条件

- API 地址格式不合法：提示用户检查格式
- API Key 无效（401）：明确提示"认证失败，请检查 API Key"
- API 地址不可达：提示"无法连接到 API 地址，请检查网络"
- 模型不存在：返回 API 原始错误信息
- 测试连接时调用一次轻量接口（如 `/v1/models`）验证连通性

### 2.2 项目文件夹初始化

**目标**：创建一个小说项目的标准文件夹结构。

#### 用户操作

1. 指定一个空文件夹路径（或新建文件夹）
2. 点击"创建项目"
3. 工具自动生成标准目录结构

#### 生成的文件夹结构

```
{项目名}/
├── novel.config.json          ← 项目配置
├── prompts/
│   ├── system.md              ← SYSTEM 级提示词（初始为空，引导用户填写）
│   └── scenes/                ← SCENE 级提示词目录
├── chapters/                  ← 章节目录
├── elements/                  ← 要素目录
│   ├── characters/
│   ├── locations/
│   ├── timeline/
│   ├── plots/
│   └── worldbuilding/
└── .novel/                    ← 工具内部数据（自动生成，用户不可见）
```

#### 边界条件

- 指定的文件夹非空：提示用户选择空文件夹，或在当前文件夹下新建子文件夹
- 文件夹路径无写入权限：提示权限错误

## 三、内容生成模块

### 3.1 生成指令

**目标**：用户通过自然语言指令驱动 AI 生成小说内容。

#### 用户操作

1. 在输入区域输入生成指令（如"写第三章，主角在密境中遇到一只受伤的灵兽"）
2. 选择当前使用的 SCENE 提示词（可选，默认使用最近的）
3. 点击发送

#### 系统行为

1. **上下文组装**：
   - 加载 SYSTEM 提示词
   - 加载当前 SCENE 提示词（如有）
   - 根据用户指令进行 RAG 检索，获取相关要素（人物、地点、情节等）
   - 组装完整上下文，发送给大模型
2. **内容生成**：AI 以流式方式生成内容
3. **实时预览**：界面实时展示 AI 正在生成的内容
4. **文件写入**：生成完成后，AI 将内容写入 `chapters/` 目录下的 .md 文件
5. **要素提取**：AI 同时从生成内容中提取要素，分别写入 `elements/` 下对应的子文件夹
6. **向量化**：新的要素文件自动进行 Embedding 向量化并存入向量库

#### 输入/输出

| 项目 | 说明 |
|:-----|:-----|
| 输入 | 用户指令（文本）、当前 SCENE 提示词（可选） |
| 输出 | 章节文件（写入 `chapters/`）、要素文件（写入 `elements/`）、向量索引更新 |
| 涉及的提示词 | SYSTEM 提示词、SCENE 提示词、要素提取专用提示词（内置） |

#### 边界条件

- 用户未配置模型：提示"请先配置模型 API"
- 未设置 SYSTEM 提示词：提示"请先完成系统提示词设置"，并提供跳转链接
- 向量库为空（首个项目首次生成）：RAG 检索跳过，仅使用 SYSTEM + SCENE + 用户指令
- 生成过程中网络断开：提示"生成中断，已生成的内容已保存"
- AI 生成的内容未包含可提取的要素：正常完成，不强制要求每次都提取要素

### 3.2 内容修改

**目标**：用户通过自然语言指令让 AI 修改已有内容。

#### 用户操作

1. 选中要修改的章节或内容片段（可选，不选则由 AI 根据指令判断）
2. 输入修改指令（如"把第三段的环境描写改得更阴暗一些"）
3. 发送

#### 系统行为

1. 读取目标文件内容
2. 执行 RAG 检索获取相关上下文
3. AI 生成修改后的完整文件内容
4. 覆盖写入原文件（Git 自动记录变更）

#### 边界条件

- 修改指令与文件内容无关：AI 应明确告知"未找到需要修改的内容"
- 用户未选中任何文件且指令模糊：AI 应询问"请指定要修改的文件，或更明确地描述修改目标"

### 3.3 实时预览

**目标**：生成/修改过程中实时展示内容变化。

#### 展示方式

- **流式输出**：AI 生成的内容逐字/逐段显示，不等待生成完毕
- **文件预览**：完成后可切换查看 .md 文件的渲染预览（类似 Markdown 阅读模式）
- **光标定位**：生成/修改完成后自动滚动到变更位置

## 四、提示词调教模块

### 4.1 SYSTEM 级提示词

**目标**：用户与 AI 通过多轮对话来编写适用于整部小说的全局提示词。

#### 用户操作

1. 打开"系统提示词"编辑页面
2. 进入对话模式，与 AI 进行多轮对话来商量提示词内容
3. 对话过程中，AI 实时生成并预览提示词效果
4. 满意后，点击"确认并保存"

#### 对话窗口交互

- **输入**：用户的自然语言描述（如"我写的是玄幻小说，偏热血风格，用第三人称叙事"）
- **AI 行为**：根据用户描述逐步构建提示词，每次修改后展示完整提示词预览
- **历史版本**：保存提示词修改历史，用户可回退到之前的版本

#### 存储

- 最终确认的提示词保存至 `prompts/system.md`

#### 边界条件

- 提示词为空时生成内容：提示"请先设置系统提示词"
- 用户在对话中要求 AI 生成内容而非编写提示词：AI 应引导回提示词编写

### 4.2 SCENE 级提示词

**目标**：用户为特定章节或情节段落设定场景提示词。

#### 用户操作

1. 创建新的 SCENE 提示词（自动命名如 `scene-001`，用户可重命名）
2. 进入对话模式，与 AI 商量该场景的设定
3. 确认后保存

#### 场景提示词内容示例

- 本章节的核心冲突和目标
- 需要推进的剧情线
- 氛围基调
- 需要登场的人物及其当前状态
- 与上一章节的衔接要求

#### 存储

- 保存至 `prompts/scenes/{scene-name}.md`
- 生成内容时可选择使用哪个 SCENE 提示词

#### 边界条件

- 未设置任何 SCENE 提示词：生成时不附带 SCENE 提示词，仅使用 SYSTEM + RAG + 用户指令

## 五、AI 校对模块

### 5.1 校对触发

**目标**：用户选择要校对的章节，让 AI 检查一致性和准确性。

#### 用户操作

1. 进入校对页面
2. 选择要校对的章节（可多选），未选择时默认选择最后 N 章
3. 选择校对方向（可多选）：
   - **要素 → 章节**：根据要素数据检查章节是否有遗漏或矛盾
   - **章节 → 要素**：根据章节内容检查要素数据是否需要更新
   - **一致性检查**：检查前后章节是否有逻辑矛盾

### 5.2 校对执行

#### 系统行为

1. AI 读取目标章节内容 + 相关要素
2. 按选定方向逐章检查
3. 生成校对报告，每条问题包含：
   - **位置**：具体文件和内容位置
   - **类型**：遗漏 / 矛盾 / 建议修改
   - **描述**：问题的具体说明
   - **建议**：AI 建议的修改方案
4. 用户可逐条查看并决定：采纳 / 忽略 / 修改建议后采纳

#### 输入/输出

| 项目 | 说明 |
|:-----|:-----|
| 输入 | 选择的章节、校对方向 |
| 输出 | 校对报告（问题列表 + 建议方案） |
| 后续操作 | 用户确认修改后，AI 执行文件变更 |

### 5.3 校对执行与版本管理

**目标**：校对修改在独立分支进行，不影响主内容。

#### 系统行为

1. 用户确认修改后，自动创建新分支 `proofread/{timestamp}`
2. 在新分支上执行所有采纳的修改
3. 自动 Git commit，提交信息包含校对摘要
4. 用户可在校对页面查看主分支与新分支的差异
5. 确认无误后，合并到主分支；不满意则丢弃新分支

#### 边界条件

- 校对过程中 API 调用失败：已完成的校对结果保留，支持断点续做
- 校对报告为空（未发现问题）：提示"未发现需要修改的问题"
- 同一章节正在进行上一次校对的分支：提示"上次校对结果尚未处理，请先处理后再发起新校对"
- AI 建议的修改可能篡改故事情节：校对提示词中严格约束 AI 只能修正事实性错误和逻辑问题，不能擅自修改剧情走向

## 六、AI 章节整理模块

### 6.1 整理触发

**目标**：将散乱的章节文件规范化整理。

#### 用户操作

1. 进入章节整理页面
2. 选择要整理的文件范围（可多选），或全选
3. 选择/确认整理规范（命名规则、字数范围、拆分/合并策略等）
   - 整理规范由用户通过与 AI 对话来约定（首次使用时引导用户完成）
   - 已有规范可直接复用

### 6.2 整理执行

#### 系统行为

1. AI 读取目标文件内容
2. 按整理规范生成整理方案，包括：
   - **拆分**：哪些文件需要拆分成多个文件
   - **合并**：哪些文件需要合并为一个文件
   - **重命名**：文件名变更对照表
   - **内容调整**：拆分/合并处的衔接内容补写
3. 展示整理预览（类似 diff 视图），用户可逐项确认或驳回
4. 用户确认后，AI 执行文件操作
5. 变更自动提交到 Git

#### 输入/输出

| 项目 | 说明 |
|:-----|:-----|
| 输入 | 选中的文件、整理规范 |
| 输出 | 整理方案（预览）、执行后的文件变更 |

#### 边界条件

- 选中的文件被外部工具锁定（如 Obsidian 正在编辑）：提示"文件被占用，请关闭后重试"
- 整理方案中某个文件的修改与外部最新修改冲突：提示冲突，让用户选择保留哪个版本
- 整理规范未设定：引导用户先通过 AI 对话完成规范设定

## 七、版本管理模块

### 7.1 自动保存

**目标**：每次 AI 操作后自动记录变更。

#### 触发条件

- AI 生成/修改章节内容后
- AI 提取/更新要素后
- 用户手动保存操作后

#### 系统行为

- 自动 `git add` + `git commit`
- 提交信息自动生成，格式：`{操作类型}: {简要描述}`，如：
  - `generate: 生成第003章-密境遇险.md`
  - `edit: 修改第001章-初入江湖.md 中的环境描写`
  - `element: 更新人物-林远.md 的状态信息`
  - `proofread: 校对第001-003章，修复3处逻辑矛盾`

### 7.2 分支管理

#### 分支策略

| 分支类型 | 命名规则 | 用途 |
|:---------|:---------|:-----|
| 主分支 | `main` | 正式内容 |
| 校对分支 | `proofread/{YYYYMMDD-HHmmss}` | AI 校对产生的修改 |
| 备份分支 | `backup/{备注}` | 用户手动创建的存档点 |

#### 用户操作

- **创建备份**：输入备份名称，从当前 main 分支创建备份分支
- **查看分支**：列出所有分支，显示创建时间和备注
- **切换分支**：切换到指定分支查看历史内容
- **合并校对分支**：校对确认后，将 proofread 分支合并到 main
- **丢弃校对分支**：校对结果不满意，删除 proofread 分支

#### 边界条件

- 合并校对分支时有文件冲突：提示用户手动解决冲突（不自动合并）
- 备份分支过多：提示用户清理旧备份

## 八、非功能需求

### 8.1 性能要求

- RAG 检索延迟：从用户发送指令到 AI 开始生成，应控制在 5 秒以内（不含 AI 生成时间）
- 文件写入：AI 生成完成后，内容应在 1 秒内写入文件
- 界面响应：UI 交互操作应在 200ms 内响应

### 8.2 兼容性

- **模型兼容**：支持所有兼容 OpenAI Chat Completions API 格式的模型（`/v1/chat/completions`）
- **文件兼容**：章节和要素文件为标准 Markdown 格式，可被 Obsidian、Typora、VS Code 等任意 Markdown 工具直接打开
- **编辑器复用**：用户可使用任意第三方 Markdown 编辑器直接编辑项目文件夹中的文件，工具应能检测到外部变更

### 8.3 数据安全

- API Key 存储在本地项目配置文件中，不上传到任何服务器
- 项目数据完全存储在用户本地文件系统中
