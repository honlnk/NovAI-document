## 二、核心依赖选型

### 2.1 向量检索：Orama

> [!important] 选型决策
> 使用 **Orama** 作为向量检索引擎，替代最初规划的 SQLite + 向量数据库方案。

#### 选型依据

| 考量 | Orama 的情况 |
|:-----|:-------------|
| 实现方式 | 纯 JavaScript，无 WASM 依赖 |
| 包体积 | ~15-30KB（gzip） |
| 向量维度 | 不限制（由外部 Embedding API 决定） |
| 外部 Embedding 兼容 | ✅ 自行管理向量，与任意 Embedding API 配合 |
| 元数据过滤 | ✅ 支持 filter 机制（按 `last_updated_chapter`、`type` 等字段过滤） |
| 持久化 | ✅ 内置 IndexedDB 持久化插件 |
| 混合搜索 | ✅ 原生支持 BM25 全文检索 + 向量语义检索 |
| 性能 | 千到万级数据量表现良好，满足小说场景 |
| 许可证 | Apache 2.0 |
| 社区 | ~12k+ stars，文档完善，活跃维护 |

#### 对比过的方案

| 方案 | 未选择原因 |
|:-----|:-----------|
| LanceDB | WASM 包体积大（数 MB），Rust 核心增加复杂度，性能优势在万级数据下不明显 |
| sqlite-vec | 需要额外引入 SQLite WASM，构建配置复杂，且暴力搜索无高级索引 |
| Vectra | 功能过于简单，社区较小 |

#### 关键变化

产品规划中原设计为 "SQLite + 向量数据库" 双层存储。选择 Orama 后，**不再需要 SQLite**，存储架构简化为：

```
Orama (IndexedDB) ← 向量索引
文件系统 (.md/.json) ← 正式数据（真相来源）
isomorphic-git ← Git 版本管理（自有 IndexedDB 后端）
```

#### 在项目中的使用方式

```typescript
import { create, insert, search } from '@orama/orama'
import { persist } from '@orama/plugin-persisted'

// 1. 创建索引（每个小说项目一个实例）
const db = await create({
  schema: {
    id: 'string',
    projectId: 'string',
    type: 'string',           // characters / locations / timeline / plots / worldbuilding
    name: 'string',
    summary: 'string',
    retrievalText: 'string',
    vector: 'vector[1536]',   // 维度由 Embedding 模型决定
    lastUpdatedChapter: 'string',
    relatedChapters: 'string[]',
    tags: 'string[]',
    sourcePath: 'string',
  },
})

// 2. 从 Embedding API 获取向量后插入
await insert(db, {
  id: 'char-linyuan',
  projectId: 'novel-20260401-001',
  type: 'characters',
  name: '林远',
  summary: '主角，性格沉稳，当前正在密境历练，处于突破前夜。',
  retrievalText: '类型：人物\n名称：林远\n摘要：主角，性格沉稳，当前正在密境历练，处于突破前夜。',
  vector: [0.12, -0.34, ...],  // 来自 Embedding API
  lastUpdatedChapter: '第049章',
  relatedChapters: ['第001章', '第003章', '第010章'],
  tags: ['主角', '剑修'],
  sourcePath: 'elements/characters/林远.md',
})

// 3. RAG 粗检索
const results = await search(db, {
  term: '林远突破瓶颈',
  vector: {
    value: queryVector,       // 用户指令的向量
    property: 'vector',
  },
  limit: 20,                   // 由项目设置控制
})
```

> [!note] 补充说明
> Orama 在 NovAI 中负责“粗召回”，默认精排层由独立的 `Rerank` 模型承担。索引文档除向量本体外，还需要保存 `retrievalText`、`summary`、`sourcePath` 等检索元数据，以支持过滤、解释、增量更新与重建。

### 2.2 Git 集成：isomorphic-git

#### 选型依据

| 考量 | isomorphic-git 的情况 |
|:-----|:----------------------|
| 实现方式 | 纯 JavaScript，无 WASM/Native 依赖 |
| 版本 | v1.37.4（2026-03-13 发布） |
| 维护状态 | ✅ 活跃（2026-03-16 最近提交） |
| Stars | 8,129 |
| 支持的操作 | init, add, commit, branch, checkout, merge, log, status, fetch, push, pull 等 |
| 不支持的操作 | rebase（我们不需要）, submodule, git-lfs |
| 存储后端 | 自定义 `fs` 适配器（IndexedDB / OPFS / File System Access API） |
| Tree-shaking | ✅ 支持按需导入，控制打包体积 |

#### 功能覆盖

| 我们的需求 | API | 状态 |
|:-----------|:----|:-----|
| 自动 git add + commit | `add` + `commit` | ✅ 完全支持 |
| 创建/切换/删除分支 | `branch` + `checkout` + `deleteBranch` | ✅ 完全支持 |
| 列出分支 | `listBranches` | ✅ 完全支持 |
| 查看 commit 历史 | `log` | ✅ 完全支持 |
| 合并分支 | `merge` + `fastForward` | ✅ 完全支持 |
| 查看文件差异 | `statusMatrix` + `readTree` + `readBlob` | ⚠️ 需组合 API 封装 |
| 回退到指定版本 | `checkout`（指定 commit SHA） | ⚠️ 无原生 `reset --hard`，可通过 checkout 实现 |

#### 需要自行封装的功能

1. **Diff 封装**：isomorphic-git 没有直接的 `git diff` API，需要通过 `readTree` + `readBlob` 对比两个 commit 的文件树，逐文件生成 diff。建议封装为 `utils/gitDiff.ts`。
2. **File System Access API 适配器**：isomorphic-git 通过 `fs` 适配器操作文件系统。需要编写一个适配器，将 `showDirectoryPicker()` 返回的目录句柄映射为 isomorphic-git 需要的 `readFile`、`writeFile`、`readdir`、`stat`、`mkdir`、`unlink` 等接口。建议封装为 `adapters/fsAccessAdapter.ts`。

#### 对比过的方案

| 方案 | 未选择原因 |
|:-----|:-----------|
| wasm-git (libgit2 WASM) | 版本 0.0.x，API 不稳定，社区小（796 stars） |
| gitkit-js | 已停止维护（2018 年最后更新） |
| git-html5.js | 已停止维护（2021 年最后更新） |

### 2.3 文件访问：File System Access API

#### 选型依据

纯前端架构下，NovAI 需要直接读写用户本地文件系统中的小说项目文件夹。File System Access API 是浏览器提供的标准 API，允许网页应用在获得用户授权后读写本地目录。

| 特性 | 说明 |
|:-----|:-----|
| API 触发 | `showDirectoryPicker()` 打开目录选择器 |
| 读取文件 | `FileSystemFileHandle.getFile()` |
| 写入文件 | `FileSystemFileHandle.createWritable()` |
| 遍历目录 | `FileSystemDirectoryHandle.values()` |
| 监听变更 | ❌ 无原生文件变更监听，需通过轮询 `getFile()` 的 lastModified 时间戳实现 |
| 权限持久化 | ✅ 首次授权后，同一域名下的后续访问浏览器可能自动放行（无需重复确认） |

#### 浏览器兼容性

| 浏览器 | 支持情况 |
|:-------|:---------|
| Chrome / Edge 86+ | ✅ 完全支持 |
| Firefox / Safari | ❌ 不支持 |
| 移动端浏览器 | ❌ 不支持 |

> [!warning] 兼容性限制
> File System Access API 仅 Chromium 内核浏览器支持。这是纯前端方案的固有限制。
> 后续通过 Tauri 打包为桌面应用后，此限制将消除（Tauri 可直接操作本地文件系统）。

#### 外部编辑器协作

由于项目文件以标准 .md 和 .json 格式存储在用户本地文件系统中，用户可以随时使用 Obsidian、Typora、VS Code 等工具直接打开编辑。工具需要检测外部变更（轮询 `lastModified`）并在检测到变更时更新 UI。

## 三、存储架构

### 3.1 数据分层

```
┌─────────────────────────────────────────────────────────┐
│                     用户可见层                           │
│                                                          │
│  .md 文件（章节、要素、提示词）← 真相来源（Source of Truth）│
│  .json 文件（项目配置）    ← 真相来源（Source of Truth） │
│                                                          │
├─────────────────────────────────────────────────────────┤
│                     工具内部层                           │
│                                                          │
│  Orama 索引（IndexedDB）  ← 向量检索加速（可重建缓存）   │
│  isomorphic-git（IndexedDB）← Git 版本数据               │
│  Pinia + localStorage     ← UI 状态和用户偏好            │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**核心原则**：
- **.md 和 .json 是真相来源**（Source of Truth），人类可读可编辑
- **Orama 索引是可重建的缓存**，删除后重新扫描 .md 文件并调用 Embedding API 即可恢复
- **isomorphic-git 管理自己的 Git 数据**，与文件系统操作解耦

### 3.2 各存储职责

| 存储方式 | 存什么 | 谁写入 | 谁读取 |
|:---------|:-------|:-------|:-------|
| **.md 文件** | 章节正文、要素数据、提示词 | AI 生成后写入 | 用户 + AI |
| **novel.config.json** | 模型配置、项目参数 | 用户在设置页修改 | AI 功能模块 |
| **Orama (IndexedDB)** | 要素的 Embedding 向量索引与检索元数据 | AI 提取要素后自动向量化 | RAG 粗检索模块 |
| **isomorphic-git** | Git 对象（commits、trees、blobs） | 每次 AI 操作后自动 commit | 版本管理模块 |
| **Pinia** | UI 状态（当前打开的文件、视图模式等） | UI 交互时更新 | Vue 组件 |
| **localStorage** | 用户偏好（侧边栏展开状态等） | 用户操作时更新 | 应用启动时读取 |

### 5.3 对话上下文压缩

当对话 Token 达到上限 80% 时，将较早的消息发送给 AI 生成摘要：

```typescript
// core/llm/compressor.ts
async function compressMessages(messages: Message[], keepCount: number): Promise<CompressedResult> {
  const toCompress = messages.slice(0, -keepCount)
  const toKeep = messages.slice(-keepCount)

  const summary = await llm.chat([
    { role: 'system', content: '请将以下对话内容压缩为结构化摘要...' },
    ...toCompress,
  ])

  return { summary, remaining: toKeep }
}
```

### 5.4 外部文件变更检测

由于 File System Access API 不支持文件变更监听，采用轮询方案：

```typescript
// utils/watcher.ts
function startWatching(dirHandle: FileSystemDirectoryHandle, intervalMs = 3000) {
  const fileCache = new Map<string, number>() // path → lastModified

  setInterval(async () => {
    for await (const entry of dirHandle.values()) {
      if (entry.kind === 'file') {
        const file = await entry.getFile()
        const cached = fileCache.get(entry.name)
        if (cached !== file.lastModified) {
          // 文件被外部修改，触发更新
          emit('file-changed', entry.name)
          fileCache.set(entry.name, file.lastModified)
        }
      }
    }
  }, intervalMs)
}
```

## 七、技术风险与缓解

| 风险 | 影响 | 缓解措施 |
|:-----|:-----|:---------|
| File System Access API 仅 Chromium 支持 | Firefox/Safari 用户无法使用 | 阶段一接受此限制；后续 Tauri 桌面端消除 |
