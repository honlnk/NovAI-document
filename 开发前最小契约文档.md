## 一、本文档解决什么问题

当前设计文档已经回答了这些问题：

- 产品要做什么

本文档用于补齐这些最小实现契约。

## 二、当前开发边界

### 2.1 本阶段明确要做的

本阶段只覆盖以下主链路：

1. 创建或打开一个本地小说项目
2. 读取并展示项目文件树
3. 预览 Markdown 文件
4. 配置 `LLM` 与 `Embedding`
5. 输入创作指令并生成章节
6. 将章节写入 `chapters/`
7. 从章节中提取要素并写入 `elements/`
8. 为要素建立 Embedding 索引
9. 下一次生成时执行 RAG 粗检索并复用相关要素

### 2.2 本阶段明确不做的

以下内容允许保留接口或占位结构，但不要求在第一批代码中完成：

- AI 精筛选
- Git 自动提交与版本管理界面
- 校对模块
- 章节整理模块
- 提示词调教对话工作流
- 外部文件变更监听优化
- 跨平台桌面端适配

## 三、统一目录结构

### 3.1 代码仓库目录结构

`v1` 代码结构以 [v1技术选型决策](v1技术选型决策.md) 为准，采用下面这版作为首批落地结构：

```txt
src/
├── app/                 # 应用级入口、路由、providers
├── views/               # 首页 / 工作区 / 设置页
├── layouts/             # 大型布局
├── components/          # 通用基础组件
├── features/            # 按业务拆分的前端模块
│   ├── project/
│   ├── file-tree/
│   ├── preview/
│   ├── chat/
│   └── settings/
├── stores/              # Pinia 状态
├── core/                # 与框架无关的核心逻辑
│   ├── fs/
│   ├── llm/
│   ├── rag/
│   ├── markdown/
│   └── project/
├── types/
└── utils/
```

### 3.2 小说项目目录结构

小说项目目录初始化时，统一生成以下结构：

```txt
{项目名}/
├── novel.config.json
├── prompts/
│   ├── system.md
│   └── scenes/
├── chapters/
├── elements/
│   ├── characters/
│   ├── locations/
│   ├── timeline/
│   ├── plots/
│   └── worldbuilding/
└── .novel/
    └── manifest.json
```

说明：

- `.novel/` 用于工具内部数据
- `manifest.json` 在 `MVP` 阶段先只保存最基础的内部元信息
- 向量索引仍保存在 IndexedDB，不直接写入项目目录

## 四、项目配置契约

### 4.1 `novel.config.json` 最小结构

首版统一采用以下 JSON 结构：

```json
{
  "version": 1,
  "project": {
    "name": "我的第一本小说",
    "createdAt": "2026-04-01T00:00:00.000Z",
    "updatedAt": "2026-04-01T00:00:00.000Z"
  },
  "llm": {
    "baseUrl": "",
    "apiKey": "",
    "model": ""
  },
  "embedding": {
    "baseUrl": "",
    "apiKey": "",
    "model": ""
  },
  "settings": {
    "generationRecentChapters": 3,
    "ragCandidateLimit": 20,
    "proofreadDefaultChapters": 3,
    "organizeDefaultChapters": 10,
    "conversationTokenLimit": 12000,
    "compressionKeepRecentTurns": 5
  }
}
```

### 4.2 字段约束

| 字段 | 类型 | 说明 |
|:-----|:-----|:-----|
| `version` | `number` | 配置版本号，首版固定为 `1` |
| `project.name` | `string` | 项目名称，展示用 |
| `project.createdAt` | `string` | ISO 时间字符串 |
| `project.updatedAt` | `string` | ISO 时间字符串 |
| `llm.baseUrl` | `string` | LLM API 地址 |
| `llm.apiKey` | `string` | LLM Key，允许为空字符串但视为未配置 |
| `llm.model` | `string` | 模型名，可为空 |
| `embedding.baseUrl` | `string` | Embedding API 地址 |
| `embedding.apiKey` | `string` | Embedding Key |
| `embedding.model` | `string` | Embedding 模型名 |
| `settings.generationRecentChapters` | `number` | 生成时携带的近期章节数 |
| `settings.ragCandidateLimit` | `number` | 粗检索返回条数 |
| `settings.proofreadDefaultChapters` | `number` | 为后续模块预留 |
| `settings.organizeDefaultChapters` | `number` | 为后续模块预留 |
| `settings.conversationTokenLimit` | `number` | 对话 Token 上限 |
| `settings.compressionKeepRecentTurns` | `number` | 压缩时保留的最近轮数 |

### 4.3 配置处理规则

- 所有读取 `novel.config.json` 的地方必须先做 schema 校验
- 缺失字段时使用默认值补齐
- `version` 不匹配时，先进入迁移层，再输出标准结构
- `apiKey` 在 UI 中默认掩码展示
- 配置保存采用整文件覆盖写入，不做局部 patch

## 五、文件格式契约

### 5.1 章节文件命名规则

`MVP` 统一采用：

```txt
第{三位数字序号}章-{章节标题}.md
```

示例：

```txt
第001章-初入江湖.md
第002章-密境遇险.md
```

### 5.2 章节文件最小格式

章节文件正文不强制 YAML Frontmatter。
`MVP` 阶段统一采用“标题 + 正文”的简单结构：

```md
# 第001章 初入江湖

正文内容……
```

约束：

- 第一行必须是一级标题
- 文件正文按 Markdown 普通文本处理
- 不要求 AI 在 `MVP` 阶段生成复杂元数据

### 5.3 要素文件命名规则

每个要素文件对应一个“稳定实体”或“稳定主题”，命名优先使用中文可读名称：

```txt
elements/characters/林远.md
elements/locations/黑风谷.md
elements/plots/密境试炼.md
```

如同名冲突，追加短标识：

```txt
林远-主角.md
林远-前朝将军.md
```

### 5.4 要素文件最小格式

`MVP` 阶段统一采用 YAML Frontmatter + Markdown 正文：

```md
---
id: char-linyuan
type: character
name: 林远
summary: 主角，性格沉稳，习剑多年，当前正在密境历练。
last_updated_chapter: 第004章
related_chapters:
  - 第001章
  - 第004章
tags:
  - 主角
  - 剑修
---

林远出身寒门，早年长期习剑。

当前状态：
- 已进入密境
- 正面临突破瓶颈
```

### 5.5 要素类型枚举

`MVP` 阶段固定支持五类：

| 目录名 | `type` 值 |
|:-------|:----------|
| `characters` | `character` |
| `locations` | `location` |
| `timeline` | `timeline` |
| `plots` | `plot` |
| `worldbuilding` | `worldbuilding` |

### 5.6 要素写入规则

- AI 每次提取要素时，优先更新已有文件，而不是无条件新建重复文件
- 无法判断是否为已有实体时，允许先新建文件，但必须写入 `summary`
- `last_updated_chapter` 必须更新为当前章节
- `related_chapters` 必须去重
- 要素正文允许自由文本，但必须能被后续 Embedding 使用

## 六、内部数据契约

### 6.1 `.novel/manifest.json`

首版最小结构如下：

```json
{
  "version": 1,
  "projectId": "novel-20260401-001",
  "createdAt": "2026-04-01T00:00:00.000Z",
  "lastOpenedAt": "2026-04-01T00:00:00.000Z"
}
```

说明：

- `projectId` 用于 IndexedDB 中区分不同项目的索引命名空间
- 首版不在 manifest 中保存复杂缓存状态

### 6.2 IndexedDB 中的 Orama 文档结构

向量索引中的文档统一至少包含：

```ts
type IndexedElementDocument = {
  id: string
  projectId: string
  type: 'character' | 'location' | 'timeline' | 'plot' | 'worldbuilding'
  name: string
  content: string
  vector: number[]
  lastUpdatedChapter: string
  relatedChapters: string[]
  sourcePath: string
}
```

约束：

- `content` 用于 Embedding 和检索
- `sourcePath` 必须能回溯到真实要素文件
- IndexedDB 文档不是事实来源，只是检索缓存

## 七、AI 协议契约

### 7.1 总体原则

`MVP` 阶段不要让 UI 直接消费“自由文本 + 猜结构”。
所有需要进入业务层的数据，都必须经过明确协议或解析层转换。

### 7.2 LLM 客户端输出协议

流式生成阶段，`core/llm/client` 对上层统一暴露以下事件：

```ts
type LlmStreamEvent =
  | { type: 'start' }
  | { type: 'delta'; text: string }
  | { type: 'finish'; text: string }
  | { type: 'error'; message: string }
```

约束：

- UI 层只关心流事件，不直接处理原始 API chunk
- `finish.text` 必须是完整拼接后的最终文本

### 7.3 章节生成返回结构

章节生成完成后，业务层应输出：

```ts
type ChapterGenerationResult = {
  chapterFileName: string
  chapterTitle: string
  markdown: string
}
```

要求：

- `markdown` 必须是可直接写入章节文件的完整内容
- `chapterFileName` 由业务层负责标准化

### 7.4 要素提取返回结构

要素提取统一要求模型返回 JSON，再由解析层校验：

```ts
type ExtractedElement = {
  id: string
  type: 'character' | 'location' | 'timeline' | 'plot' | 'worldbuilding'
  name: string
  summary: string
  body: string
  relatedChapters: string[]
  tags: string[]
}
```

```ts
type ElementExtractionResult = {
  elements: ExtractedElement[]
}
```

约束：

- JSON 解析失败时，整次提取视为失败，但不影响章节生成主链路成功
- `elements` 可为空数组
- 进入文件写入层前必须补充 `last_updated_chapter`

### 7.5 RAG 粗检索输入输出

粗检索输入：

```ts
type RagSearchInput = {
  projectId: string
  query: string
  limit: number
}
```

粗检索输出：

```ts
type RagSearchHit = {
  id: string
  type: string
  name: string
  content: string
  score: number
  lastUpdatedChapter: string
  relatedChapters: string[]
  sourcePath: string
}
```

```ts
type RagSearchResult = {
  hits: RagSearchHit[]
}
```

`MVP` 阶段只实现粗检索，不引入 AI 精筛选。

## 八、模块边界契约

### 8.1 允许依赖浏览器 API 的层

以下模块允许直接接触浏览器文件系统能力：

- `core/fs/`
- `features/project/`
- 极少量应用初始化代码

### 8.2 不允许依赖浏览器 API 的层

以下模块必须保持可移植，不直接调用 `showDirectoryPicker()` 等 API：

- `core/llm/`
- `core/rag/`
- `core/project/`
- `stores/`
- `features/chat/`

原因：

- 后续迁移 Tauri 时可以最小化重写范围
- 核心逻辑可单独测试

### 8.3 推荐职责划分

| 模块 | 职责 | 不负责什么 |
|:-----|:-----|:-----------|
| `core/fs` | 目录扫描、读写文件、项目初始化 | 不负责页面状态 |
| `core/project` | 解析项目结构、装配项目元信息 | 不直接访问组件 |
| `core/llm` | API 调用、流式输出适配 | 不负责任何文件写入 |
| `core/rag` | Embedding、索引、检索 | 不负责 UI 展示 |
| `features/project` | 创建/打开项目的前端流程 | 不实现底层读写细节 |
| `features/file-tree` | 文件树展示与交互 | 不处理 Markdown 渲染 |
| `features/preview` | 文件预览与生成预览 | 不处理项目初始化 |
| `features/chat` | 用户输入、消息列表、生成状态 | 不直接操作 File System API |

## 十、错误处理与降级约定

`MVP` 阶段先统一以下原则：

1. 章节生成成功时，即使要素提取失败，也算主链路成功。
2. 要素提取成功但 Embedding 失败时，要素文件仍然保留，索引标记为待补建。
3. RAG 检索失败时，允许直接使用“提示词 + 近期章节 + 用户指令”继续生成。
4. 模型测试连接失败时，必须返回可展示给用户的明确信息，不允许只抛原始异常对象。
5. 配置文件格式不合法时，项目允许进入“受限打开”状态，并提示修复，而不是直接让整个应用崩溃。

## 十一、开发顺序最终确认

基于本文档，第一批代码开发顺序明确为：

1. 升级工程基线：TypeScript、Vue Router、Pinia、基础目录结构
2. 首页、工作区、设置页骨架
3. 项目创建与打开
4. 文件树与文件预览
5. `novel.config.json` 读写与模型配置
6. 对话输入区与章节生成流式链路
7. 要素提取与写入
8. Embedding 与 Orama 粗检索

这条顺序原则上不再反复摇摆。
