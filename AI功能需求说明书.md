# NovAI（诺艾）- 需求说明书

> [!info] 本文档定位
> 详细描述 NovAI 各功能模块的具体需求，包括输入输出、交互逻辑、边界条件等，供开发团队对齐理解。
> 产品规划文档见 [产品规划](NovAI产品规划.md)。

## 一、概述

### 1.1 用户角色

NovAI 不设用户系统，使用者即作者。一个文件夹就是一个小说项目，不同项目之间数据完全隔离。

### 1.2 使用场景

| 场景    | 描述                           |
| :---- | :--------------------------- |
| 新建项目  | 创建一个小说项目文件夹，配置模型、设定系统提示词     |
| 日常创作  | 输入指令让 AI 生成内容，AI 自动写入文件并提取要素 |
| 提示词调教 | 与 AI 对话来编写和优化系统提示词 / 场景提示词   |
| 内容校对  | 让 AI 检查章节一致性和要素准确性           |
| 章节整理  | 让 AI 按规范对散乱的章节文件进行拆分、合并、重命名  |

---

## 二、项目配置需求

### 2.1 模型配置

**目标**：让用户分别接入 LLM（大语言模型）、Embedding（向量化模型）和 Rerank（重排序模型）的 API 服务。

> [!important] 设计决策
> LLM、Embedding 和 Rerank **必须独立配置**，不共用同一套 API 地址和 Key。原因：
> 1. 三者的服务提供商可能不同
> 2. 即使是同一厂商，三类模型的接口行为和计费方式也不同
> 3. 独立配置更符合“生成 / 向量化 / 重排序”三层职责分离
> 4. 独立配置避免用户误操作导致其中一层不可用

#### 2.1.1 LLM 配置

**用途**：内容生成、上下文筛选、要素提取、对话交互、上下文压缩等所有 AI 对话场景。

##### 用户操作

1. 填写以下信息：
   - **API 地址**（必填）：LLM 服务的 Base URL
   - **API Key**（必填）：认证密钥
   - **模型名称**（可选）：指定使用的模型，不填则使用 API 默认模型
2. 点击"测试连接"验证配置是否可用
3. 保存配置

##### 输入/输出

| 项目 | 说明 |
|:-----|:-----|
| 输入 | API 地址、API Key、模型名称 |
| 输出 | 配置保存成功 / 连接测试结果（成功/失败 + 原因） |
| 存储 | `novel.config.json` 中的 `llm` 字段 |

##### 边界条件

- API 地址格式不合法：提示用户检查格式
- API Key 无效（401）：明确提示"认证失败，请检查 API Key"
- API 地址不可达：提示"无法连接到 API 地址，请检查网络"
- 模型不存在：返回 API 原始错误信息
- 测试连接时调用一次轻量接口（如 `/v1/models`）验证连通性

#### 2.1.2 Embedding 配置

**用途**：将要素文件向量化，存入向量库以支持 RAG 语义检索。

##### 用户操作

1. 填写以下信息：
   - **API 地址**（必填）：Embedding 服务的 Base URL
   - **API Key**（必填）：认证密钥
   - **模型名称**（可选）：指定使用的 Embedding 模型，不填则使用 API 默认模型
2. 点击"测试连接"验证配置是否可用
3. 保存配置

##### 输入/输出

| 项目 | 说明 |
|:-----|:-----|
| 输入 | API 地址、API Key、模型名称 |
| 输出 | 配置保存成功 / 连接测试结果（成功/失败 + 原因） |
| 存储 | `novel.config.json` 中的 `embedding` 字段 |

##### 边界条件

- 与 LLM 配置的边界条件相同
- Embedding 模型不支持中文：提示"当前 Embedding 模型可能不支持中文，向量化结果可能不准确"

#### 2.1.3 Rerank 配置

**用途**：对 RAG 粗检索得到的候选要素进行二次排序，默认作为 NovAI 的精排层。

##### 用户操作

1. 填写以下信息：
   - **是否启用**：默认关闭，开启后参与生成链路
   - **API 地址**（必填）：Rerank 服务的 Base URL
   - **API Key**（必填）：认证密钥
   - **模型名称**（必填）：指定使用的 Rerank 模型
   - **模式**（可选）：`text` 或 `multimodal`，首版默认 `text`
   - **重排保留条数**（可选）：Top N，默认 8
2. 点击"测试连接"验证配置是否可用
3. 保存配置

##### 输入/输出

| 项目 | 说明 |
|:-----|:-----|
| 输入 | 是否启用、API 地址、API Key、模型名称、模式、Top N |
| 输出 | 配置保存成功 / 连接测试结果（成功/失败 + 原因） |
| 存储 | `novel.config.json` 中的 `rerank` 字段 |

##### 边界条件

- 与 LLM 配置的边界条件相同
- 用户启用了 Rerank 但未填写模型名：提示"请先填写 Rerank 模型名称"
- Rerank 服务不可用：自动降级为仅使用 RAG 粗检索结果继续生成

### 2.2 项目文件夹初始化

**目标**：创建一个小说项目的标准文件夹结构。

#### 用户操作

1. 指定一个空文件夹路径（或新建文件夹）
2. 点击"创建项目"
3. 工具自动生成标准目录结构

#### 生成的文件夹结构

```
{项目名}/
├── novel.config.json          ← 项目配置
├── prompts/
│   ├── system.md              ← SYSTEM 级提示词（初始为空，引导用户填写）
│   └── scenes/                ← SCENE 级提示词目录
├── chapters/                  ← 章节目录
├── elements/                  ← 要素目录
│   ├── characters/
│   ├── locations/
│   ├── timeline/
│   ├── plots/
│   └── worldbuilding/
└── .novel/                    ← 工具内部数据（自动生成，用户不可见）
```

#### 边界条件

- 指定的文件夹非空：提示用户选择空文件夹，或在当前文件夹下新建子文件夹
- 文件夹路径无写入权限：提示权限错误

### 2.3 项目设置

**目标**：提供统一的设置页面，让用户自定义各功能模块的默认行为参数。

#### 用户操作

1. 进入项目设置页面
2. 查看并修改各项参数
3. 保存设置（修改后立即生效）

#### 可配置参数

| 参数 | 说明 | 默认值 | 影响的模块 |
|:-----|:-----|:-------|:-----------|
| **校对默认章节数** | 未手动选择时，自动选取最后 N 章进行校对 | 3 | AI 校对模块 |
| **整理默认章节数** | 未手动选择时，自动选取最后 N 个章节进行整理 | 10 | AI 章节整理模块 |
| **生成上下文章节数** | 生成内容时，自动携带最后 N 章的原文作为近期上下文（确保叙事连贯） | 3 | 内容生成模块 |
| **RAG 粗检索返回条数** | 向量数据库粗检索阶段返回的候选要素数量（宁多勿少，后续由 Rerank 精排） | 20 | 内容生成模块 |
| **Rerank 保留条数** | Rerank 二次排序后最终进入上下文组装的候选要素数量 | 8 | 内容生成模块 |
| **对话上下文 Token 上限** | 对话窗口中聊天记录的最大 Token 数，接近上限时自动触发压缩（详见对话上下文压缩策略） | 视模型而定 | 所有对话窗口 |
| **压缩保留轮数** | 压缩对话时保留最近 N 轮原始对话不被压缩，仅压缩更早的记录 | 5 | 对话上下文管理 |

#### 存储

- 所有参数保存在 `novel.config.json` 中的 `settings` 字段

#### 边界条件

- 用户将某项参数设为 0 或负数：重置为该项的默认值，并提示用户
- 用户修改设置后未保存就离开页面：提示"有未保存的更改，是否保存？"

## 三、内容生成模块

### 3.1 生成指令

**目标**：用户通过自然语言指令驱动 AI 生成小说内容。

#### 用户操作

1. 在输入区域输入生成指令（如"写第三章，主角在密境中遇到一只受伤的灵兽"）
2. 选择当前使用的 SCENE 提示词（可选，默认使用最近的）
3. 点击发送

#### 系统行为

1. **上下文组装**（两阶段检索 + 近期章节）：
   - 加载 SYSTEM 提示词
   - 加载当前 SCENE 提示词（如有）
   - **加载近期章节**：读取 `chapters/` 目录下最后 N 章的原文内容（N 由项目设置中的"生成上下文章节数"控制），作为近期叙事上下文
   - **粗检索**：根据用户指令进行 RAG 语义检索，从向量库中获取相似度 Top K 的候选要素（宁多勿少）
   - **精排序**：如已配置 Rerank，则将候选要素 + 用户指令发送给 Rerank 模型进行二次排序，保留 Top N 的更相关要素
   - **高级审校（可选）**：在高质量模式下，可将 Rerank 结果再交给 LLM 做一次语义审校，过滤极少量仍然不适合进入当前情节的候选
   - 组装近期章节 + 精排后的要素 + 提示词 + 用户指令，发送给大模型生成内容
2. **内容生成**：AI 以流式方式生成内容
3. **实时预览**：界面实时展示 AI 正在生成的内容
4. **文件写入**：生成完成后，AI 将内容写入 `chapters/` 目录下的 .md 文件
5. **要素提取**：AI 同时从生成内容中提取要素，分别写入 `elements/` 下对应的子文件夹
6. **向量化**：新的要素文件自动进行 Embedding 向量化并存入向量库

#### 输入/输出

| 项目     | 说明                                               |
| :----- | :----------------------------------------------- |
| 输入     | 用户指令（文本）、当前 SCENE 提示词（可选）                        |
| 输出     | 章节文件（写入 `chapters/`）、要素文件（写入 `elements/`）、向量索引更新 |
| 涉及的提示词 | SYSTEM 提示词、SCENE 提示词、要素提取专用提示词（内置），以及高级审校模式下的上下文审校提示词（内置） |

#### 边界条件

- 用户未配置模型：提示"请先配置模型 API"
- 未设置 SYSTEM 提示词：提示"请先完成系统提示词设置"，并提供跳转链接（用户可强行放弃设置/使用SYSTEM提示词）
- 向量库为空（首个项目首次生成）：RAG 粗检索和 Rerank 均跳过，仅使用 SYSTEM + SCENE + 近期章节 + 用户指令
- Rerank 未配置：直接使用 RAG 粗检索的原始结果继续生成
- Rerank 调用失败（网络异常等）：降级为直接使用 RAG 粗检索的原始结果，并在界面提示"上下文重排未生效，可能包含非相关要素"
- 高级审校模式下 LLM 审校结果为空：正常继续，仅使用 SYSTEM + SCENE + 近期章节 + 用户指令生成，不强制要求必须有要素上下文
- 生成过程中网络断开：提示"生成中断，已生成的内容已保存"
- AI 生成的内容未包含可提取的要素：正常完成，不强制要求每次都提取要素
- 已有章节数不足 N（如设置 N=3 但目前只有 1 章）：正常继续，加载所有已有章节，不报错

### 3.2 内容修改

**目标**：用户通过自然语言指令让 AI 修改已有内容。

#### 用户操作

1. 选中要修改的章节或内容片段（可选，不选则由 AI 根据指令判断）
2. 输入修改指令（如"把第三段的环境描写改得更阴暗一些"）
3. 发送

#### 系统行为

1. 读取目标文件内容
2. 执行 RAG 检索获取相关上下文
3. AI 生成修改后的完整文件内容
4. 覆盖写入原文件（Git 自动记录变更）

#### 边界条件

- 修改指令与文件内容无关：AI 应明确告知"未找到需要修改的内容"
- 用户未选中任何文件且指令模糊：AI 应询问"请指定要修改的文件，或更明确地描述修改目标"

### 3.3 实时预览

**目标**：生成/修改过程中实时展示内容变化。

#### 展示方式

- **流式输出**：AI 生成的内容逐字/逐段显示，不等待生成完毕
- **文件预览**：完成后可切换查看 .md 文件的渲染预览（类似 Markdown 阅读模式）
- **光标定位**：生成/修改完成后自动滚动到变更位置

## 四、提示词调教模块

### 4.1 SYSTEM 级提示词

**目标**：用户与 AI 通过多轮对话来编写适用于整部小说的全局提示词。

#### 用户操作

1. 打开"系统提示词"编辑页面
2. 进入对话模式，与 AI 进行多轮对话来商量提示词内容
3. 对话过程中，AI 实时生成并预览提示词效果
4. 满意后，点击"确认并保存"

#### 对话窗口交互

- **输入**：用户的自然语言描述（如"我写的是玄幻小说，偏热血风格，用第三人称叙事"）
- **AI 行为**：根据用户描述逐步构建提示词，每次修改后展示完整提示词预览
- **历史版本**：对话过程中，每次 AI 迭代生成新版本时，将上一版本存入内存中的版本列表，用户可在列表中前后跳转或从某个历史版本重新分支编辑（仅限当前编辑会话，关闭页面后版本列表清空）

#### 存储

- 最终确认的提示词保存至 `prompts/system.md`

#### 边界条件

- 提示词为空时生成内容：不强制拦截，但给出一次提醒"当前未设置系统提示词，AI 生成风格可能不受控制"，用户可选择继续生成或去设置

### 4.2 SCENE 级提示词

**目标**：用户为特定章节或情节段落设定场景提示词。

#### 用户操作

1. 创建新的 SCENE 提示词（自动命名如 `scene-001`，用户可重命名）
2. 进入对话模式，与 AI 商量该场景的设定
3. 确认后保存

#### 场景提示词内容示例

- 本章节的核心冲突和目标
- 需要推进的剧情线
- 氛围基调
- 需要登场的人物及其当前状态
- 与上一章节的衔接要求

#### 存储

- 保存至 `prompts/scenes/{scene-name}.md`
- 生成内容时可选择使用哪个 SCENE 提示词

#### 边界条件

- 未设置任何 SCENE 提示词：生成时不附带 SCENE 提示词，仅使用 SYSTEM + 近期章节 + RAG + 用户指令

## 五、AI 校对模块

### 5.1 校对触发

**目标**：用户选择要校对的章节，让 AI 检查一致性和准确性。

#### 用户操作

1. 进入校对页面
2. 选择要校对的章节（可多选），未选择时默认选择最后 N 章（用户可在设置页中设置）
3. 选择校对方向（可多选）：
   - **要素 → 章节**：根据要素数据检查章节是否有遗漏或矛盾
   - **章节 → 要素**：根据章节内容检查要素数据是否需要更新
   - **一致性检查**：检查前后章节是否有逻辑矛盾

### 5.2 校对执行

#### 系统行为

1. AI 读取目标章节内容 + 相关要素
2. 按选定方向逐章检查
3. 生成校对报告，每条问题包含：
   - **位置**：具体文件和内容位置
   - **类型**：遗漏 / 矛盾 / 建议修改
   - **描述**：问题的具体说明
   - **建议**：AI 建议的修改方案
4. 用户可逐条查看并决定：采纳 / 忽略 / 修改建议后采纳

#### 输入/输出

| 项目 | 说明 |
|:-----|:-----|
| 输入 | 选择的章节、校对方向 |
| 输出 | 校对报告（问题列表 + 建议方案） |
| 后续操作 | 用户确认修改后，AI 执行文件变更 |

### 5.3 校对执行与版本管理

**目标**：校对修改在独立分支进行，不影响主内容。

#### 系统行为

1. 用户确认修改后，自动创建新分支 `proofread/{timestamp}`
2. 在新分支上执行所有采纳的修改
3. 自动 Git commit，提交信息包含校对摘要
4. 用户可在校对页面查看主分支与新分支的差异
5. 确认无误后，合并到主分支；不满意则丢弃新分支

#### 边界条件

- 校对过程中 API 调用失败：已完成的校对结果保留，支持断点续做
- 校对报告为空（未发现问题）：提示"未发现需要修改的问题"
- 同一章节正在进行上一次校对的分支：提示"上次校对结果尚未处理，请先处理后再发起新校对"
- AI 建议的修改可能篡改故事情节：校对提示词中严格约束 AI 只能修正事实性错误和逻辑问题，不能擅自修改剧情走向

## 六、AI 章节整理模块

### 6.1 整理触发

**目标**：将散乱的章节文件规范化整理。

#### 用户操作

1. 进入章节整理页面
2. 选择要整理的文件范围（可多选），或全选，不选择就默认选择最后N个章节（可在设置页面进行配置）
3. 选择/确认整理规范（命名规则、字数范围、拆分/合并策略等）
   - 整理规范由用户通过与 AI 对话来约定（首次使用时引导用户完成）
   - 已有规范可直接复用

### 6.2 整理执行

#### 系统行为

1. AI 读取目标文件内容
2. 按整理规范生成整理方案，包括：
   - **拆分**：哪些文件需要拆分成多个文件
   - **合并**：哪些文件需要合并为一个文件
   - **重命名**：文件名变更对照表
   - **内容调整**：拆分/合并处的衔接内容补写
3. 展示整理预览（类似 diff 视图），用户可逐项确认或驳回
4. 用户确认后，AI 执行文件操作
5. 变更自动提交到 Git

#### 输入/输出

| 项目 | 说明 |
|:-----|:-----|
| 输入 | 选中的文件、整理规范 |
| 输出 | 整理方案（预览）、执行后的文件变更 |

#### 边界条件

- 选中的文件被外部工具锁定（如 Obsidian 正在编辑）：提示"文件被占用，请关闭后重试"
- 整理方案中某个文件的修改与外部最新修改冲突：提示冲突，让用户选择保留哪个版本
- 整理规范未设定：引导用户先通过 AI 对话完成规范设定

## 七、版本管理模块

### 7.1 自动保存

**目标**：每次 AI 操作后自动记录变更。

#### 触发条件

- AI 生成/修改章节内容后
- AI 提取/更新要素后
- 用户手动保存操作后

#### 系统行为

- 自动 `git add` + `git commit`
- 提交信息自动生成，格式：`{操作类型}: {简要描述}`，如：
  - `generate: 生成第003章-密境遇险.md`
  - `edit: 修改第001章-初入江湖.md 中的环境描写`
  - `element: 更新人物-林远.md 的状态信息`
  - `proofread: 校对第001-003章，修复3处逻辑矛盾`

### 7.2 分支管理

#### 分支策略

| 分支类型 | 命名规则 | 用途 |
|:---------|:---------|:-----|
| 主分支 | `main` | 正式内容 |
| 校对分支 | `proofread/{YYYYMMDD-HHmmss}` | AI 校对产生的修改 |
| 备份分支 | `backup/{备注}` | 用户手动创建的存档点 |

#### 用户操作

- **创建备份**：输入备份名称，从当前 main 分支创建备份分支
- **查看分支**：列出所有分支，显示创建时间和备注
- **切换分支**：切换到指定分支查看历史内容
- **合并校对分支**：校对确认后，将 proofread 分支合并到 main
- **丢弃校对分支**：校对结果不满意，删除 proofread 分支

#### 边界条件

- 合并校对分支时有文件冲突：提示用户手动解决冲突（不自动合并）
- 备份分支过多：提示用户清理旧备份

## 八、对话上下文管理

> [!info] 适用范围
> 本章所述的对话上下文管理策略适用于**所有对话窗口**（提示词调教、内容修改、校对对话、整理对话等），**不适用于内容生成流程**（内容生成走 RAG 上下文组装，不依赖对话历史）。

### 8.1 压缩策略

**目标**：对话记录过长时，自动或手动压缩历史消息，防止上下文溢出。

#### 压缩机制

| 项目 | 说明 |
|:-----|:-----|
| **Token 上限** | 由项目设置中的"对话上下文 Token 上限"控制 |
| **自动触发** | 当对话 Token 数达到上限的 80% 时，自动触发压缩 |
| **手动触发** | 对话窗口提供"压缩上下文"按钮，用户可随时点击 |
| **压缩方式** | 将较早的对话记录发送给 AI 生成摘要，用摘要替换原始记录 |
| **保留规则** | 最近 N 轮对话原文保留不被压缩，仅压缩更早的记录 |

#### 压缩流程

1. 触发压缩（自动或手动）
2. 将超出保留范围的历史消息发送给 AI
3. AI 生成一段结构化摘要，包含：已讨论的关键决策、当前进度、待确认的事项
4. 用摘要替换原始消息，保留最近 N 轮对话原文
5. 界面提示"对话上下文已压缩"，用户可点击查看压缩后的摘要内容

#### 用户感知

- 压缩后，较早的对话消息在界面上折叠为一条摘要消息（可展开查看）
- 用户仍可正常继续对话，AI 基于摘要 + 保留的近期对话继续工作
- 压缩不丢失关键信息，但细节程度会有所降低

#### 边界条件

- 压缩过程中 API 调用失败（网络异常）：取消本次压缩，保持原始对话不变，提示"上下文压缩失败，将在下次发送时重试"
- 对话记录未达到自动触发阈值且用户未手动触发：不压缩，保持完整对话历史
- 首次对话（记录过少）：不触发压缩，保留所有原始消息

## 九、非功能需求

### 9.1 性能要求

- RAG 检索延迟：从用户发送指令到 AI 开始生成，应控制在 5 秒以内（不含 AI 生成时间）
- 文件写入：AI 生成完成后，内容应在 1 秒内写入文件
- 界面响应：UI 交互操作应在 200ms 内响应

### 9.2 兼容性

- **模型兼容**：支持所有兼容 OpenAI Chat Completions API 格式的 LLM（`/v1/chat/completions`），以及兼容 OpenAI Embeddings API 格式的向量化模型（`/v1/embeddings`）
- **文件兼容**：章节和要素文件为标准 Markdown 格式，可被 Obsidian、Typora、VS Code 等任意 Markdown 工具直接打开
- **编辑器复用**：用户可使用任意第三方 Markdown 编辑器直接编辑项目文件夹中的文件，工具应能检测到外部变更

### 9.3 数据安全

- API Key 存储在本地项目配置文件中，不上传到任何服务器
- 项目数据完全存储在用户本地文件系统中
