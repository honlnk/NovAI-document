# NovAI（诺艾）- 向量索引与重排序设计

> [!info] 本文档定位
> 本文档用于补充 NovAI 在 RAG 场景下的向量索引、Embedding 文本组装、候选召回、Rerank 二次排序、索引失效与重建，以及用户可见解释层设计。
>
> 产品定位见 [产品规划](NovAI产品规划.md)，总体技术选型见 [技术架构设计](技术架构设计.md)，AI 功能流程见 [AI功能需求说明书](AI功能需求说明书.md)。

> [!important] 本文档的核心决策
> NovAI 的默认检索链路调整为：
> `要素文件 -> Embedding -> Orama 粗召回 -> Rerank 二次排序 -> LLM 生成`
>
> 其中：
> - `.md / .json` 文件始终是真相来源（Source of Truth）
> - IndexedDB 中的向量索引始终是可重建缓存
> - Rerank 默认作为“精排序层”，不再默认由通用 LLM 直接承担候选筛选主职责
> - LLM 仍保留为高级审校层和兜底层，而不是被完全移除

---

## 一、存储分层

### 1.1 设计目标

NovAI 的向量数据存储方案需要同时满足以下目标：

- 对用户透明，不要求用户理解向量数据库
- 与本地文件系统协同，允许用户手动编辑 `.md` 文件
- 索引可重建，不让缓存成为“唯一数据来源”
- 支持按项目隔离，避免不同小说项目混淆
- 支持后续接入 Rerank、多模态召回、索引迁移与模型切换

### 1.2 三层存储结构

NovAI 中与 RAG 相关的数据，分为三层：

| 层级 | 存储位置 | 存什么 | 是否真相来源 |
|:-----|:---------|:-------|:-------------|
| **内容层** | 项目目录下的 `elements/`、`chapters/`、`prompts/`、`novel.config.json` | 小说章节、要素文件、提示词、配置 | ✅ 是 |
| **索引层** | 浏览器 `IndexedDB` | 要素向量、检索文本、检索元数据、索引状态 | ❌ 否 |
| **运行层** | 内存状态（Pinia / 运行时缓存） | 当前查询、召回结果、重排结果、上下文草稿 | ❌ 否 |

### 1.3 为什么不把向量直接写回项目目录

不建议把 Embedding 向量直接写回 `.md` 或 `.json` 文件，原因如下：

- 向量数组体积大，可读性极差，会污染用户文件
- 向量和具体模型强绑定，模型一旦切换就会整体过期
- 向量本质是检索缓存，不属于作者真正关心的创作资产
- 后续如果调整 Embedding 文本模板，旧向量也需要整体失效

因此，项目目录中只保留“可重建所需的原始事实”，不保留原始向量值。

### 1.4 IndexedDB 中建议分为两类数据

IndexedDB 中建议至少维护两组逻辑对象：

1. `element_documents`
2. `index_meta`

职责如下：

| 对象 | 作用 |
|:-----|:-----|
| `element_documents` | 存储每个要素对应的检索文本、向量与检索元数据 |
| `index_meta` | 存储当前项目索引的状态、模型信息、版本信息、重建时间、错误信息 |

### 1.5 项目级隔离规则

每一个小说项目都必须拥有独立的索引命名空间。

推荐规则：

- 业务层以 `projectId` 作为逻辑隔离键
- 同一个 IndexedDB 实例中可以存多个项目的文档，但所有查询必须带 `projectId`
- 也允许为每个项目创建独立数据库名，二者选其一即可，但整个项目内必须统一

`MVP` 阶段更推荐第一种：

- 一个数据库实例
- 多项目共存
- `projectId` 强隔离

优点是结构更简单，便于后续迁移和调试。

## 二、IndexedDB 数据结构

### 2.1 `element_documents` 最小结构

每一个参与检索的要素，在索引层建议至少落一条文档：

```ts
type IndexedElementDocument = {
  id: string
  projectId: string
  sourcePath: string
  type: 'character' | 'location' | 'timeline' | 'plot' | 'worldbuilding'
  name: string
  summary: string
  retrievalText: string
  vector: number[]
  lastUpdatedChapter: string
  relatedChapters: string[]
  tags: string[]
  sourceModifiedAt: string
  indexedAt: string
  contentHash: string
  embeddingProvider: string
  embeddingModel: string
  embeddingDim: number
  embeddingTextVersion: number
}
```

### 2.2 为什么必须同时存元数据

必须存，而且这是 RAG 项目里非常关键的一步。

如果只存：

- `id`
- `content`
- `vector`

那么短期可以完成“最原始的相似度搜索”，但很快会遇到以下问题：

- 无法判断索引是否已经过期
- 无法按要素类型过滤
- 无法做“近期优先”的排序
- 无法向用户解释“为什么检索到这条”
- 无法在模型切换后判断是否需要重建
- 无法在内容被外部编辑后增量更新

因此，元数据不是可选增强，而是检索系统的一部分。

### 2.3 元数据字段建议分组

建议把元数据按用途理解为四组：

#### A. 回溯类

- `id`
- `projectId`
- `sourcePath`

用途：

- 回到真实要素文件
- 区分项目
- 保证同一要素可被稳定更新而不是重复插入

#### B. 检索类

- `type`
- `name`
- `summary`
- `retrievalText`
- `vector`
- `tags`

用途：

- 召回
- 重排
- 展示检索结果摘要

#### C. 排序类

- `lastUpdatedChapter`
- `relatedChapters`

用途：

- 做新鲜度加权
- 做“与近期章节关系更紧密”的增强排序
- 为后续冲突处理提供上下文

#### D. 维护类

- `sourceModifiedAt`
- `indexedAt`
- `contentHash`
- `embeddingProvider`
- `embeddingModel`
- `embeddingDim`
- `embeddingTextVersion`

用途：

- 判断是否需要增量更新
- 判断当前向量是否仍与配置兼容
- 触发全量重建

### 2.4 `index_meta` 最小结构

项目级索引元信息建议如下：

```ts
type ProjectIndexMeta = {
  projectId: string
  status: 'empty' | 'building' | 'ready' | 'stale' | 'rebuilding' | 'error'
  documentCount: number
  embeddingProvider: string
  embeddingModel: string
  embeddingDim: number
  embeddingTextVersion: number
  rerankProvider?: string
  rerankModel?: string
  lastBuildAt?: string
  lastFullRebuildAt?: string
  lastError?: string
}
```

### 2.5 是否要额外存“章节级向量”

当前阶段不建议。

NovAI 当前设计是“AI 提取要素后，以要素为天然检索块”，而不是传统文档 RAG 的“对正文强行切 chunk”。因此：

- 默认只对 `elements/` 下的要素建索引
- 不对整章正文直接建立主检索索引
- 后续如需做“章节片段召回”，应单独设计第二套索引，不与要素索引混用

这样更符合 NovAI 的产品定位，也更容易解释给用户。

## 三、Embedding 文本组装规范

### 3.1 设计原则

向量质量很大程度取决于“送给 Embedding 模型的文本长什么样”。

因此，NovAI 不能简单把整篇要素文件原样送去向量化，而应定义统一的 `retrievalText` 组装规则。

设计原则如下：

- 让文本尽量包含实体身份
- 保留当前状态，而不是只保留背景故事
- 保留少量结构化标签，帮助模型理解语义
- 不把无意义元数据堆满文本
- 保持格式稳定，便于后续重建

### 3.2 默认组装模板

建议默认模板如下：

```txt
类型：{type_label}
名称：{name}
摘要：{summary}
标签：{tags_joined}
最后更新章节：{last_updated_chapter}
相关章节：{related_chapters_joined}

正文：
{body}
```

示例：

```txt
类型：人物
名称：林远
摘要：主角，性格沉稳，当前正在密境历练，处于突破前夜。
标签：主角、剑修、密境
最后更新章节：第004章
相关章节：第001章、第004章

正文：
林远出身寒门，早年长期习剑。
当前状态：
- 已进入密境
- 正面临突破瓶颈
```

### 3.3 为什么不用“只嵌入 summary”

只嵌入 `summary` 会过度压缩信息，容易导致以下问题：

- 检索结果不稳定
- 同名实体更难区分
- 当前状态与历史状态混在一起时，信息不够
- Rerank 候选质量下降

因此建议：

- `summary` 用于轻量展示
- `retrievalText` 用于真正 Embedding

二者不要混为同一个字段。

### 3.4 文本截断与清洗规则

建议增加以下规则：

- 空字段不输出字段名，避免模板噪音
- `tags` 最多保留前 10 个
- `relatedChapters` 最多保留最近 20 个
- `body` 过长时优先保留“当前状态”与最近补充内容
- 删除纯格式噪音，如多余空行、重复标题、无意义占位文本

### 3.5 版本化规则

`retrievalText` 模板必须带版本号。

建议：

- 初版记为 `embeddingTextVersion = 1`
- 只要模板字段或拼接顺序有显著变化，就升级版本号
- 当版本号变化时，视为当前项目索引需要全量重建

## 四、Orama 召回策略

### 4.1 Orama 在 NovAI 中的角色

Orama 在 NovAI 中的定位不是“用户单独安装的本地工具”，而是应用内嵌的检索引擎依赖。

它负责：

- 存储向量索引
- 根据 query vector 做相似度召回
- 根据 metadata 做过滤
- 在需要时做简单混合检索

它不负责：

- 生成 Embedding
- 替代 LLM 写作
- 直接成为用户操作的独立数据库软件

换句话说，用户使用的是 NovAI；Orama 只是 NovAI 内部的一部分。

### 4.2 默认召回流程

默认召回流程建议如下：

1. 用户输入生成指令
2. 系统组装检索 query
3. 使用 Embedding 模型将 query 向量化
4. 在 Orama 中按 `projectId` 检索 Top K 候选
5. 对候选结果做轻量后处理
6. 将候选送入 Rerank 层

### 4.3 默认召回参数

`MVP` 阶段建议：

- `TopK = 20`
- 仅检索当前项目
- 默认不过滤要素类型
- 允许后续按 Scene 或任务类型增加过滤条件

### 4.4 召回时建议支持的过滤条件

Orama 层建议预留这些过滤能力：

- `projectId`
- `type`
- `lastUpdatedChapter`
- `tags`

示例场景：

- 场景主要发生在“地点切换”时，可提高 `location` 的召回优先级
- 用户明确要求“补完人物状态”，可优先召回 `character`

### 4.5 召回阶段只做“宁多勿少”

召回层的任务不是一次性决定最终上下文，而是尽量把真正相关的候选捞出来。

因此：

- 召回层允许带入一定噪音
- 不应在召回层做过重的规则判断
- 真正的精细排序和裁剪交给 Rerank 层完成

### 4.6 是否启用混合检索

`MVP` 阶段建议默认以向量召回为主，不强制启用 BM25 + 向量的混合检索。

原因：

- 当前要素文件大多较短，语义向量价值更高
- 混合检索会增加调参复杂度
- 当前最核心的问题不是“搜不到字面匹配”，而是“语义相似但剧情无关”

后续如果发现：

- 专有名词召回不稳定
- 多个同类实体混淆

再评估开启混合检索即可。

## 五、Rerank 二次排序策略

### 5.1 为什么要引入 Rerank

原设计中，Orama 粗召回之后由通用 LLM 直接做精筛选。这种方式可行，但不是最优默认方案。

引入 Rerank 后，默认链路改为：

`Orama TopK 候选 -> Rerank 重排序 -> 取 TopN -> LLM 生成`

好处如下：

- 成本通常低于额外一次 LLM 精筛选
- 输出更稳定，便于程序消费
- 延迟更低，更适合高频生成
- 更符合标准 RAG 系统的常见分层

### 5.2 在 NovAI 中的职责定位

Rerank 层负责：

- 接收查询文本
- 接收候选列表
- 输出候选排序结果
- 产出更适合进入生成上下文的 TopN

它不负责：

- 生成最终正文
- 替代 Embedding 生成向量
- 替代要素提取

### 5.3 默认输入结构

建议对 Rerank 提供统一输入：

```ts
type RerankInput = {
  query: string
  candidates: Array<{
    id: string
    type: string
    name: string
    summary: string
    retrievalText: string
    lastUpdatedChapter: string
    relatedChapters: string[]
  }>
  topN: number
}
```

`MVP` 阶段可以只把 `retrievalText` 送入模型；业务层保留其他字段用于回填和解释。

### 5.4 Rerank 是否可以替代原来的 LLM 精筛选

可以作为默认平替，甚至可以成为更高优先级的主路径。

推荐策略如下：

| 模式 | 流程 | 适用场景 |
|:-----|:-----|:---------|
| **默认模式** | Orama 召回 -> Rerank -> LLM 生成 | 日常创作 |
| **兼容模式** | Orama 召回 -> 直接进入 LLM 生成 | 未配置 Rerank 时 |
| **高级模式** | Orama 召回 -> Rerank -> LLM 审校 -> LLM 生成 | 高质量模式、复杂剧情 |

因此，Rerank 不是简单“加一个模型”，而是重新定义默认精排层。

### 5.5 是否要支持多模态 Rerank

当前阶段建议先不作为默认路径。

原因：

- NovAI 当前的主检索数据是文本要素
- 多模态 Rerank 更适合图文混合候选
- 如果未来要把插图、地图、人物卡图片一起纳入检索，再引入多模态模式更自然

因此建议：

- 当前文档先把 `text rerank` 作为标准方案
- `multimodal rerank` 只做接口预留

### 5.6 Rerank 模型需要独立配置

需要。

原因如下：

- 它和 Embedding、LLM 是不同职责的模型
- 提供商、计费方式、接口地址都可能不同
- 用户可能只配置 Embedding，不配置 Rerank
- 未来用户也可能只在高质量模式下启用 Rerank

建议在 `novel.config.json` 中新增：

```json
{
  "rerank": {
    "enabled": false,
    "baseUrl": "",
    "apiKey": "",
    "model": "",
    "mode": "text",
    "topN": 8
  }
}
```

建议新增字段说明：

| 字段 | 类型 | 说明 |
|:-----|:-----|:-----|
| `enabled` | `boolean` | 是否启用 Rerank |
| `baseUrl` | `string` | Rerank API 地址 |
| `apiKey` | `string` | Rerank API Key |
| `model` | `string` | Rerank 模型名 |
| `mode` | `string` | `text` 或 `multimodal` |
| `topN` | `number` | 重排后最终保留的候选条数 |

### 5.7 Rerank 输出后的后处理

Rerank 之后还建议保留少量业务规则：

- 结果去重
- 同一实体重复状态时优先保留更新版本
- 对极低分结果做截断
- 控制最终送入生成的总条数和总字数

这一步仍属于业务层，而不是模型层。

## 六、索引失效与重建规则

### 6.1 为什么必须设计失效规则

向量索引不是“建好一次就永远能用”的静态资产。

以下任意变化，都可能让当前索引不再可靠：

- 要素文件被用户手动修改
- 要素文件被 AI 更新
- Embedding 模型切换
- Embedding 文本模板升级
- 向量维度变化
- 索引构建过程中断

因此，必须把“索引什么时候失效、何时增量更新、何时全量重建”明确写进设计。

### 6.2 索引状态定义

建议项目级索引状态如下：

| 状态 | 含义 |
|:-----|:-----|
| `empty` | 尚未建立任何要素索引 |
| `building` | 首次构建中 |
| `ready` | 索引可用 |
| `stale` | 存在部分过期文档，需要增量更新 |
| `rebuilding` | 正在做全量重建 |
| `error` | 最近一次构建失败，需要用户关注 |

### 6.3 增量更新触发条件

满足以下任一条件时，将单个要素标记为需要增量更新：

- `sourceModifiedAt` 晚于 `indexedAt`
- `contentHash` 变化
- 要素文件首次被创建
- 要素文件路径存在但索引中不存在对应 `id`

增量更新动作：

- 重新读取要素文件
- 重新生成 `retrievalText`
- 重新请求 Embedding
- 覆盖更新索引文档

### 6.4 全量重建触发条件

满足以下任一条件时，整个项目索引应视为需要全量重建：

- `embeddingModel` 变化
- `embeddingProvider` 变化
- `embeddingDim` 变化
- `embeddingTextVersion` 升级
- 索引结构迁移版本变化
- 用户手动点击“重建向量索引”

### 6.5 用户体验建议

建议在 UI 上明确反馈当前索引状态：

- 首次打开项目且索引为空：提示“尚未建立要素索引”
- 检测到部分过期：提示“部分要素索引已过期，将在后台更新”
- 检测到模型切换：提示“当前 Embedding 模型已变更，需要重建索引”
- 构建失败：提示错误原因并提供重试入口

### 6.6 构建策略建议

建议：

- 首次建索引允许后台执行
- 日常编辑优先增量更新
- 全量重建前给出明确确认，避免无感触发额外成本
- 重建过程中旧索引可继续服务，但应标记为“可能过期”

### 6.7 空库与冷启动策略

当项目尚无要素索引时：

- 允许跳过 RAG 继续生成
- 首次生成后立即触发要素提取与建索引
- 如项目已有大量历史章节，可提供“一键扫描并建立要素索引”入口

这能让新项目尽快进入“越写越准”的正循环。

## 七、检索结果解释与用户可控性

### 7.1 为什么要让用户看见检索结果

NovAI 的价值不只是“自动帮用户搜”，还要让用户逐渐建立对系统的信任。

如果用户无法知道：

- 系统为什么命中了这些要素
- 为什么某些要素没被选中
- 当前上下文主要受哪些历史事实影响

那么 RAG 会像黑盒一样难以调试，也难以被真正依赖。

### 7.2 建议展示三层结果

一次生成时，建议 UI 至少能区分三层：

| 层级 | 含义 |
|:-----|:-----|
| **召回候选** | Orama 找到的 TopK 候选 |
| **重排结果** | Rerank 重新排序后的候选 |
| **最终上下文** | 真正进入生成上下文的要素 |

### 7.3 建议展示的解释信息

每个最终命中的要素，建议可查看以下信息：

- 要素名称
- 要素类型
- 摘要
- 最后更新章节
- 命中原因摘要
- 是否经过 Rerank 提升

`MVP` 阶段不要求展示复杂分数公式，但至少要能回答用户一句话：

“这条为什么会进来？”

### 7.4 用户可控性建议

后续建议支持以下手动干预能力：

- 手动排除某条候选
- 手动强制包含某条要素
- 对当前生成临时锁定某类要素优先级

这不是为了让用户承担系统工作，而是为了在复杂剧情下保留可控性。

### 7.5 与 LLM 审校层的关系

当启用高级模式时，LLM 仍可以对 Rerank 结果做一次“上下文审校”，例如：

- 发现时间冲突
- 发现角色状态过时
- 发现多个候选语义相近但只有一个应进入上下文

此时：

- Rerank 是主排序层
- LLM 是高阶语义审校层

二者并不冲突。

## 九、当前阶段的最终建议

如果只保留一组最关键结论，建议定为以下 7 条：

1. 向量只存 IndexedDB，不写回项目文件
2. 向量索引必须同时保存检索元数据
3. 每条索引文档必须能回溯到真实要素文件
4. `retrievalText` 必须有明确模板和版本号
5. Orama 负责粗召回，不负责生成和业务判断
6. Rerank 应成为默认精排层，并支持独立配置
7. 索引必须有明确的失效、增量更新与全量重建规则
