# NovAI（诺艾）- v1 技术选型决策

> [!info] 文档定位
> 本文档用于确认 NovAI 在第一阶段落地开发时采用的技术选型，重点回答“现在先定什么”与“哪些先不定”。

## 一、决策目标

当前项目已经明确：

- 产品形态是以 AI 对话驱动的小说创作工具
- 第一阶段以浏览器端为主
- 前端框架采用 Vue

但如果不尽快补齐其余技术决策，后续开发会反复摇摆在以下问题之间：

- 是否先上 TypeScript
- 是否需要统一的状态管理和路由方案
- 是否要引入 CSS 工具链
- 是否一开始就依赖大型组件库
- 项目代码目录按“页面”拆还是按“业务能力”拆

因此，本文档的目标是确定一套适合 NovAI `v1` 的、可立即开工的前端技术基线。

## 二、v1 技术选型结论

### 2.1 核心框架

| 维度 | 选型 | 结论 |
|:-----|:-----|:-----|
| 前端框架 | Vue 3 | 采用 |
| 语言 | TypeScript | 采用 |
| 构建工具 | Vite | 采用 |
| 路由 | Vue Router | 采用 |
| 状态管理 | Pinia | 采用 |

### 2.2 UI 与样式

| 维度 | 选型 | 结论 |
|:-----|:-----|:-----|
| CSS 方案 | Tailwind CSS | 采用 |
| 图标库 | lucide-vue-next | 采用 |
| 组件策略 | 自建基础组件 + 视情况补充 Headless 组件 | 采用 |
| 大型组件库 | Element Plus / Ant Design Vue | `v1` 不采用 |

### 2.3 核心能力依赖

| 维度 | 选型 | 结论 |
|:-----|:-----|:-----|
| Markdown 渲染 | markdown-it | 采用 |
| Vue 工具库 | @vueuse/core | 采用 |
| 数据校验 | zod | 采用 |
| 向量检索 | @orama/orama | 采用 |
| Git | isomorphic-git | 采用 |

## 三、为什么这样选

### 3.1 为什么采用 TypeScript

NovAI 不是一个单页展示站，而是一个包含以下复杂对象的工作台型应用：

- 小说项目
- 章节文件
- 要素文件
- 提示词配置
- 模型配置
- RAG 检索结果
- Git 历史和差异视图

这些对象在多个模块之间会频繁流转。若继续使用纯 JavaScript，后期很容易出现：

- 字段命名不统一
- 配置结构悄悄漂移
- 文件系统返回值和 UI 状态结构混乱
- AI 调用结果与预期结构不一致时难以及早发现

因此，`v1` 就上 TypeScript，能显著降低中后期维护成本。

### 3.2 为什么采用 Vue Router

UI 文档已经明确了至少三个一级页面：

- 首页 `/`
- 主工作区 `/project/:id`
- 设置页 `/project/:id/settings`

这已经不是“一个页面切状态”能优雅承载的规模，因此路由应在 `v1` 就建立，而不是后补。

### 3.3 为什么采用 Pinia

NovAI 后续会有明显的跨组件共享状态，例如：

- 当前打开的项目
- 当前选中的文件
- 文件树展开状态
- 对话上下文状态
- 右侧内容区的视图模式
- 模型配置与项目设置

这些状态若全部放在局部组件中，主工作区很快会变得难以维护。Pinia 与 Vue 3 配合自然，足够轻量，也适合按模块逐步扩展。

### 3.4 为什么采用 Tailwind CSS

NovAI 的界面不是传统表单后台，而是典型的“创作工作台”：

- 左侧文件树
- 中间 AI 对话区
- 右侧内容面板
- 多种面板切换和响应式布局

这种产品的界面特点是布局复杂、组件形态变化多、视觉风格需要一定辨识度。相比重度依赖现成组件库，Tailwind 更适合：

- 快速搭建工作台布局
- 精细控制间距、层级、响应式和状态样式
- 在早期持续迭代视觉语言，而不被组件库默认风格绑住

结论：`Tailwind CSS` 对 NovAI 有必要，建议在 `v1` 即引入。

### 3.5 为什么不在 v1 引入大型组件库

当前不建议选用 `Element Plus` 或 `Ant Design Vue` 作为主组件体系。

主要原因：

- 它们更适合后台管理系统和表单系统
- 默认视觉气质较强，容易把产品做成“管理台感”
- NovAI 需要的核心组件更偏工作区布局和编辑协作，而不是复杂业务表单
- 后期若大量覆写样式，会增加额外维护成本

因此更适合的策略是：

- 用 Tailwind 构建整体视觉和布局
- 自己封装基础组件，如 `Button`、`Input`、`Textarea`、`Dialog`、`Drawer`、`Tabs`
- 遇到无障碍或复杂交互需求时，再局部补充 Headless 方案

### 3.6 为什么选 lucide-vue-next

NovAI 会频繁使用轻量图标来表达文件、设置、版本、对话、目录树等状态。`lucide-vue-next` 的优点是：

- 与 Vue 3 集成自然
- 风格简洁，适合工具型产品
- 体积和使用复杂度都较低

### 3.7 为什么选 markdown-it

项目大量内容本身就是 Markdown：

- 章节正文
- 提示词文件
- 要素文件

右侧内容区必须稳定渲染 Markdown，因此需要尽早确定渲染方案。`markdown-it` 足够成熟、简单，适合 `v1`。

### 3.8 为什么选 @vueuse/core

该项目未来会频繁使用浏览器能力和常见组合逻辑，例如：

- 本地存储
- 窗口尺寸监听
- 防抖、节流
- 快捷键
- 轮询与状态同步

`@vueuse/core` 可以减少大量样板代码，且与 Composition API 非常契合。

### 3.9 为什么选 zod

NovAI 会处理多种“外部输入”：

- `novel.config.json`
- 文件系统扫描结果
- 模型 API 返回数据
- AI 结构化输出

这些数据都存在不可信风险。`zod` 可用于在关键边界做运行时校验，避免错误数据进入业务层。

## 四、v1 建议依赖清单

建议的前端基础依赖如下：

```txt
vue
vue-router
pinia
@vueuse/core
tailwindcss
lucide-vue-next
markdown-it
zod
@orama/orama
isomorphic-git
```

建议的开发依赖如下：

```txt
typescript
vue-tsc
@vitejs/plugin-vue
vite
```

> [!note] 说明
> 上述清单是“第一阶段建议确定的基础盘”。随着项目推进，后续可以再按需加入测试、格式化、提交规范、单元测试与 E2E 测试工具。

## 五、v1 不急着定的内容

以下内容不建议在当前阶段过早拍板：

| 项目 | 当前结论 | 原因 |
|:-----|:---------|:-----|
| 桌面端方案 | 暂不落地，只保留 Tauri 方向 | 当前重点仍是浏览器端验证产品流程 |
| 移动端方案 | 暂不落地，只保留 Capacitor 方向 | 过早考虑移动端会分散注意力 |
| 富文本编辑器 | 不采用 | 产品本身强调“文件 + AI 修改”，不是在线排版器 |
| 重型动画库 | 暂不采用 | 初期价值有限，先把结构和流程跑通 |
| 完整 UI 组件库 | 暂不采用 | 当前更适合自建基础组件体系 |

## 六、目录结构建议

配合上述技术选型，建议代码仓库从一开始就采用以下目录结构：

```txt
src/
├── app/                 # 应用级入口、路由、providers
├── views/               # 页面级视图：首页 / 工作区 / 设置页
├── layouts/             # 大型布局：工作区三栏布局等
├── components/          # 通用基础组件
├── features/            # 按业务拆分的模块
│   ├── project/
│   ├── chat/
│   ├── file-tree/
│   ├── preview/
│   └── settings/
├── stores/              # Pinia 状态
├── core/                # 与框架无关的核心逻辑
│   ├── rag/
│   ├── llm/
│   ├── git/
│   ├── fs/
│   └── markdown/
├── types/               # 全局类型定义
└── utils/               # 通用工具函数
```

这样的好处是：

- 页面结构清晰
- UI 与核心逻辑解耦
- 后续接 File System Access API、Orama、isomorphic-git 时有稳定落点
- 不会过早把所有逻辑堆进 `components/` 或 `views/`

## 七、第一阶段开发顺序建议

基于这套技术选型，建议 `v1` 的启动顺序为：

1. 补齐 `TypeScript + Vue Router + Pinia + Tailwind CSS`
2. 建立首页、主工作区、设置页三个基础页面
3. 用 mock 数据完成文件树、对话面板、右侧内容区的工作区骨架
4. 定义项目配置、章节、要素、提示词的数据类型
5. 接入 File System Access API，支持创建/打开项目目录
6. 接入 Markdown 预览
7. 再开始逐步落地 LLM、Embedding、Orama、Git 相关能力

## 八、最终结论

NovAI 的 `v1` 技术基线建议确定为：

- `Vue 3`
- `TypeScript`
- `Vite`
- `Vue Router`
- `Pinia`
- `Tailwind CSS`
- `lucide-vue-next`
- `@vueuse/core`
- `markdown-it`
- `zod`
- `@orama/orama`
- `isomorphic-git`

其中最关键的决策有两条：

- `TypeScript` 现在就上，不后补
- `Tailwind CSS` 现在就上，但 `v1` 不引入大型组件库

这套组合足够稳定、自由度高，也最适合 NovAI 这种“AI 对话 + 文件工作区”的产品形态。
